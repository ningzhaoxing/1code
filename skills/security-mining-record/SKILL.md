# Security Mining Record Skill

当用户执行已授权的漏洞挖掘任务时，本 skill 只负责两个交付物：实时维护漏洞挖掘记录，并在任务完成后生成最终 Markdown 报告。它不决定测试方法，也不覆盖其他漏洞挖掘 skill 的判断。

在 1Code/VoidForge 中，右侧预览和下载入口的最终交付物固定是 Markdown：`漏洞挖掘报告.md`。如果同时使用其他漏洞挖掘 skill，它们可以决定测试流程、阶段和证据采集方式，但不要把 docx、pdf 或其他格式当成 1Code 的最终交付文件；完成收束时仍需写出 `漏洞挖掘报告.md`。

## 核心职责

先判断当前处于哪一个阶段：

- 任务执行中：持续更新 `漏洞挖掘记录.md`，只沉淀过程中的关键信息。
- 任务收束时：基于完整执行链路生成 `漏洞挖掘报告.md`，作为最终交付报告。
- 普通追问或纠偏：只补充必要记录，不要把每个追问都当成最终报告生成时机。

聊天最终回复不是最终 Markdown 报告。即使最终报告尚未生成，也必须正常给出面向用户的中文阶段性总结，说明当前结论、记录状态、报告状态和下一步。

## 文件位置

- 实时记录文件名固定为 `漏洞挖掘记录.md`。
- 最终报告文件名固定为 `漏洞挖掘报告.md`。
- 如果当前工作目录或 `漏洞挖掘-*` 子目录下已经存在 `漏洞挖掘记录.md`，继续使用该文件，并把同目录下的 `漏洞挖掘报告.md` 作为最终报告文件。
- 如果找不到既有记录文件，在当前工作目录创建 `漏洞挖掘记录.md`，最终报告写入同目录的 `漏洞挖掘报告.md`。
- 不要因为没有收到隐藏提示词或显式路径就跳过记录；路径约定由本 skill 自己决定。

## 实时记录

实时记录是工作过程文档，不是最终报告草稿。使用自然 Markdown，自行组织标题和段落；不要套用固定空章节，不要输出 JSON。

什么时候写入 `漏洞挖掘记录.md`：

- 用户确认测试对象、限制条件或授权边界时，记录已确认的信息和不能做的动作。
- 每次工具调用产生有价值结果时，记录工具名称、关键结论、证据文件路径或可复核短摘录。
- 出现疑似漏洞或被排除的线索时，记录受影响对象、问题描述、验证状态、风险判断、复现要点和下一步。
- 需要人工审批、用户纠偏或暂停时，记录原因、用户决定以及对后续工作的影响。
- 任务阶段性结束但尚未形成最终报告时，补齐当前结论、未确认线索和下一步计划。

不要把工具原始输出整段堆入记录，只保留能支持复核的关键证据、短摘录、文件路径或命令摘要。

## 最终报告

最终报告不是实时记录的简单复制。它必须在任务完成、用户明确要求导出/收束，或已有信息足以形成最终交付时，基于完整执行链路重新组织生成。

生成 `漏洞挖掘报告.md` 时必须综合：

- 用户原始目标、授权边界和后续纠偏。
- 执行过的工具调用、关键参数、关键结果和失败/中断原因。
- `漏洞挖掘记录.md` 中已经沉淀的阶段性证据和判断。
- 已确认问题、未确认线索、排除项、风险评级和复核建议。
- 报告读者需要看到的结论、证据路径、影响说明和下一步建议。

写入最终报告后，在最终回复中说明报告已生成，并给出报告文件路径。如果本轮还没达到生成最终报告的条件，最终回复仍需说明记录文件已更新、报告尚未生成，以及还缺什么条件。

最终回复中的文件入口必须使用可点击的 Markdown 链接，不要用反引号包裹文件名。文件在当前工作目录时使用：

- [漏洞挖掘记录.md](漏洞挖掘记录.md)
- [漏洞挖掘报告.md](漏洞挖掘报告.md)

如果文件位于子目录，链接目标改为对应相对路径，但链接文字仍保持「漏洞挖掘记录.md」和「漏洞挖掘报告.md」。
